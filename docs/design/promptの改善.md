
promptの改善
今回promptやるべきことを、私のほうで実務向けに整理すると、次の4点です。

1. **何を system に固定するか**
2. **何を user / 多輪履歴 / memory / 環境情報として分離するか**
3. **どの条件で圧縮・要約・破棄するか**
4. **その prompt が本当に最小十分かをどう検査するか**

---

# 2. あなたの AgentFlow で定義すべき prompt の層

私なら、prompt を **6層** に切ります。

---

## 第1層：Core System（固定・最小）

これは毎回ほぼ変わらない、最上位原則です。

入れるべきものは少なくていいです。

### 入れるもの

* この Agent の役割
* 成功条件
* 禁止事項
* 出力原則
* 不明時の扱い

### 入れないもの

* 個別案件の背景
* 長い業務説明
* 過去会話の詳細
* 都度変わる制約

### 例

```text
あなたはタスク実行支援エージェントである。
目的は、与えられた目的に対して、必要十分な情報のみを使い、
簡潔・正確・実行可能な出力を返すことである。

必ず以下を守ること：
- 目的から外れる説明を増やさない
- 不明な事実を推測で埋めない
- 指示・制約・利用可能情報を区別して扱う
- 出力は、実行に必要な内容を優先する
- 情報が不足する場合は、不足点を最小限で示す
```

---

## 第2層：Task System（毎回動的生成）

ここが一番重要です。
**今回の仕事は何か**を定義する層です。

ここには、今回の task ごとに変わる情報だけを入れます。

### 入れるもの

* 今回の目的
* 期待成果物
* 成功判定
* 優先順位
* 制約条件

### テンプレート

```text
今回のタスク目的:
{task_goal}

期待する成果物:
{deliverables}

優先順位:
1. {priority_1}
2. {priority_2}
3. {priority_3}

制約:
- {constraint_1}
- {constraint_2}

成功条件:
- {check_1}
- {check_2}
- {check_3}
```

### 例（あなたの今回の相談に当てると）

```text
今回のタスク目的:
AgentFlow における prompt 最適化ルールを整理し、
system / user / 多輪履歴 / memory の責務分離と、
token 爆発防止策を設計する。

期待する成果物:
- prompt 層設計
- 実装しやすいテンプレート
- 圧縮ルール
- チェックリスト

優先順位:
1. 冗長性削減
2. 正確性維持
3. 実装容易性

制約:
- prompt を長文化しすぎない
- 層ごとの責務を混在させない
- 動的情報は必要時のみ注入する

成功条件:
- どの情報をどこに置くべきか明確
- token 削減方針がある
- 品質確認方法がある
```

---

## 第3層：Runtime Context（実行時コンテキスト）

これは「今この瞬間だけ必要」な情報です。

### 入れるもの

* 現在のユーザー要求
* 添付資料の要約
* 参照すべき検索結果
* 実行対象のファイル / API / UI 状態

### 原則

**原文丸ごとを入れない。まず構造化する。**

### 悪い例

* 会話履歴を全部連結
* 長文仕様書を全文貼る
* 検索結果を全部投げる

### 良い例

```text
現在の入力:
- ユーザーは AgentFlow の prompt 最適化を求めている
- 重点は system / user / memory / 多輪会話の整理
- 目的は token 爆発を防ぎ、必要十分な指示だけを渡すこと

関連資料から抽出したポイント:
- context は段階圧縮する
- system prompt は静的+動的の合成物
- 安定再利用情報と都度取得情報を分離する
```

---

## 第4層：Conversation State（会話状態）

これは履歴そのものではなく、**履歴から抽出した状態**です。

### 入れるべきもの

* 現在何を決めたか
* 未解決論点
* 次にやること
* 変更された方針

### テンプレート

```text
会話状態:
- 確定事項:
  - {decided_1}
  - {decided_2}
- 未確定事項:
  - {open_1}
  - {open_2}
- 今回の返答で行うべきこと:
  - {next_action}
```

### 重要

多輪対話で本当に必要なのは「全履歴」ではなく、
**状態遷移の結果**です。

---

## 第5層：Memory / Profile / Preference

これは長期保持するが、毎回全部入れない層です。

### 置くべき情報

* 長期的な好み
* 出力言語
* コーディング規約
* 役割や業務背景
* 継続プロジェクトの固定前提

### 置いてはいけないもの

* 今回しか使わない一時情報
* すでに task system に書くべき情報
* 古くなりやすい細部

### 注入ルール

**必要な時だけ注入**です。
memory は DB に持っておき、毎回全部 system に入れない。

### 例

```text
適用すべき長期設定:
- 回答は日本語を優先
- 簡潔で実務的な表現を優先
- 不確実なことは断定しない
```

---

## 第6層：Tool / Environment / Capability Context

これは agent が今できることの説明です。

### 入れるもの

* 利用可能ツール
* 入出力制約
* 実行環境
* 取得可能データ種別

### テンプレート

```text
利用可能手段:
- ドキュメント検索
- ファイル読み取り
- コード実行
- 図表生成

環境制約:
- 外部アクセス可否: {yes/no}
- 書き込み可否: {yes/no}
- 実行時間上限: {limit}

行動規則:
- まず既存情報から判断する
- 必要な場合のみツールを使う
- ツール結果は要約して使う
```

# 3. あなたの目的に合う「最小十分 prompt 生成フロー」

ここが実装の中心です。
私は以下の順番にします。

---

## Step 1: タスクを分解する

まず model に聞く前に、AgentFlow 側で機械的に分解します。

### 抽出項目

* 今回の目的は何か
* 何を出力すれば完了か
* 判断に必要な情報は何か
* 不要な情報は何か
* 外部取得が必要か
* 過去履歴が必要か

### 内部用テンプレート

```text
タスク分析:
- 目的:
- 成果物:
- 必須入力:
- 参考入力:
- 不要入力:
- 必要な履歴:
- 必要な memory:
- 必要な tools:
```

---

## Step 2: 情報を分類する

取得した情報を以下に振り分けます。

* 固定ルール → Core System
* 今回の目的 → Task System
* 今回だけの材料 → Runtime Context
* 会話の決定事項 → Conversation State
* 長期嗜好 → Memory
* 実行手段 → Tool/Env

これをやるだけで、かなり prompt が痩せます。

---

## Step 3: 不要情報を落とす

ここが token 爆発防止の最重要ポイントです。

### 落としてよいもの

* すでに成功条件に吸収された説明
* 同義反復
* 例が多すぎる説明
* 古い会話の詳細
* 感情的な装飾
* model にとって不要な背景談

### 圧縮ルール

* 原文ではなく箇条書き化
* 1トピック1文
* 目的に関係ない履歴は切る
* 長文資料は「結論 / 制約 / 使用箇所」に要約

---

## Step 4: 実行前 prompt を合成する

最終 prompt は以下だけで十分です。

```text
[Core System]
[Task System]
[Conversation State]   ← 必要時のみ
[Relevant Memory]      ← 必要時のみ
[Runtime Context]
[Tool/Env Context]     ← 必要時のみ
```

---

# 4. token 爆発を防ぐための運用ルール

あなたが画像で意識している内容を、実装ルールに落とすとこうなります。

---

## ルール1：履歴は「全文保持」ではなく「状態化」

### NG

* 直近20往復をそのまま渡す

### OK

```text
これまでの合意:
- system prompt は固定層と動的層に分ける
- memory は必要時のみ注入する
- 今回はテンプレート設計までを行う
```

---

## ルール2：長文資料は「引用」ではなく「抽出」

### NG

* 仕様書全文を埋め込む

### OK

```text
仕様から必要な点:
- 出力は JSON 互換構造
- 最大 3 ステップで返す
- 不確実なら unknown を返す
```

---

## ルール3：memory は always-on にしない

memory を全部 system に入れると重くなります。
なので **relevance gating** が必要です。

### 例

* 言語設定 → 常時 relevant
* 特定案件の過去議論 → 案件一致時のみ relevant
* 個人の細かな嗜好 → 出力影響時のみ relevant

---

## ルール4：圧縮は段階的に行う

画像中央の考え方を、そのまま運用ルールにできます。

### 4段圧縮案

1. **Recent Window**
   直近数ターンは生で保持
2. **Structured Summary**
   少し古い履歴は決定事項・論点に要約
3. **Topic Collapse**
   古い詳細はトピック別に1行化
4. **Long-term Memory 化**
   継続価値のあるものだけ memory に昇格

---

## ルール5：目的に対して「情報の資格審査」をする

各情報に対して、投入前にこう判定します。

* この情報が無いと失敗するか
* 今回だけ必要か、長期的に必要か
* 事実か、推論か、嗜好か
* system に置くべきか、それ以外か

---

# 5. 実際に使える prompt テンプレート

以下はそのまま AgentFlow に組み込みやすい形で書きます。

---

## 5.1 Core System Prompt

```text
あなたはタスク遂行型アシスタントである。
目的は、与えられた目的に対して、必要十分な情報のみを使い、
正確で簡潔かつ実行可能な回答を返すことである。

行動原則:
- 目的から外れる説明を増やさない
- ルール、事実、推定、提案を区別する
- 情報不足時は推測で埋めず、不足点を明示する
- 出力は、実務上そのまま使える形を優先する
- 既存の制約と成功条件を優先する
```

---

## 5.2 Task Prompt Generator 用テンプレート

```text
今回のタスクを整理せよ。

出力項目:
- 目的
- 完了条件
- 必須情報
- あれば有用な情報
- 不要情報
- 必要なツール
- 必要な履歴
- 必要な memory
- 推奨出力形式

制約:
- 冗長な説明を入れない
- 今回の目的に直接関係するものだけ抽出する
```

---

## 5.3 Runtime Prompt

```text
今回の目的:
{goal}

求める成果物:
{deliverables}

必須条件:
- {must_1}
- {must_2}

参照情報:
- {fact_1}
- {fact_2}

会話上の確定事項:
- {state_1}
- {state_2}

使える手段:
- {tool_1}
- {tool_2}

出力要求:
- まず結論
- 次に実行内容
- 必要なら最後に注意点
```

---

## 5.4 多輪対話圧縮 Prompt

```text
以下の会話履歴を、次回応答に必要な情報だけに圧縮せよ。

抽出対象:
- 確定事項
- 未解決事項
- 次に行うこと
- 継続すべき制約
- 破棄してよい詳細

出力形式:
- 確定事項:
- 未解決事項:
- 次アクション:
- 継続制約:
- 破棄候補:
```

---

## 5.5 Memory 抽出 Prompt

```text
以下の情報のうち、長期的に再利用価値があるものだけを抽出せよ。

保存対象:
- 長期的な好み
- 出力言語
- 固定ルール
- 継続プロジェクトの前提

保存しない対象:
- 一時的タスク情報
- 今回限りの条件
- 古くなりやすい詳細

出力形式:
- 保存する memory
- 保存しない理由
```

---

## 5.6 最終回答の自己点検 Prompt

```text
生成した回答を自己点検せよ。

確認項目:
1. 目的に直接答えているか
2. 不要な説明が混ざっていないか
3. 必須条件を満たしているか
4. 推測を事実のように書いていないか
5. 出力がそのまま使える形か
6. もっと短くしても意味が落ちないか

問題があれば、内容を削って改善せよ。
```

# 6. 「今回のタスクは何か、どうやるか、有用情報は何か、どう検査するか」を AI に判断させる prompt

これはあなたがまさに欲しかったものだと思います。
この 1 本を前段の planner として置くとかなり強いです。

```text
あなたの役割は、回答そのものを作る前に、
今回の依頼に対して必要十分な prompt 構成を設計することである。

まず以下を判断せよ。
- 今回のタスクは何か
- 何を出力すれば完了か
- どう進めるのが最短か
- 何の情報が本当に必要か
- 何の情報は不要か
- 過去履歴は必要か
- 長期 memory は必要か
- 外部ツールや検索は必要か
- 回答後に何をチェックすべきか

出力形式:
1. タスク定義
2. 実行方針
3. 必須情報
4. 補助情報
5. 不要情報
6. 使用すべき context 層
   - system
   - task
   - runtime
   - conversation state
   - memory
   - tools/env
7. 完了判定
8. 回答前チェック項目

制約:
- 冗長な説明は禁止
- 今回の依頼に直接効く内容だけを残す
- 情報は役割別に分離する
```

# 7. 実務で一番効く設計原則

ここはかなり大事です。
prompt 最適化で本当に効くのは、書き方の上手さより **設計** です。

## 原則1

**system に何でも書かない**
system は憲法、task は作戦、runtime は現場情報です。

## 原則2

**履歴は会話ではなく状態として持つ**
多輪が長くなるほど、この差が効きます。

## 原則3

**memory は倉庫であって、常時表示板ではない**
必要な時だけ取り出す。

## 原則4

**資料は原文投入ではなく、意味抽出して入れる**
ここをやらないと token 爆発が止まりません。

## 原則5

**最後に必ず「削れるか」を見る**
良い prompt は、長い prompt ではなく、
**削っても壊れない prompt** です。

# 8. あなた向けの結論

今回、あなたが作るべき AgentFlow はこうです。

### 入力前処理

* ユーザー依頼をタスク分解
* 情報を6層に分類
* 不要情報を削除
* 必要なら履歴を圧縮
* 必要なら memory を relevance 判定して注入

### 実行 prompt

* Core System
* Task System
* Conversation State
* Relevant Memory
* Runtime Context
* Tool/Env Context

### 実行後

* 自己点検
* 冗長削減
* 完了条件一致確認
