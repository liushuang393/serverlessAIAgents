## 改善要求の要約

### 1. プロンプト中心の構成から、実行基盤中心の構成へ改善

現状の課題は、オーケストレーターや業務プロンプトに責務を寄せすぎると、復旧、再実行、監査、評価、権限制御が難しくなる点です。
そのため、以下を一等オブジェクトとして明示的に設計する必要があります。

* Session
* Artifact
* Decision
* Checkpoint
* Feedback
* Action Log

つまり、**「賢いプロンプト」で頑張る構成ではなく、「制御可能な実行基盤」に改善すること**が要求されています 

---

### 2. コンテキスト投入方法の改善

全履歴・全ナレッジ・全顧客情報を毎回そのまま渡すのではなく、**階層化されたコンテキスト管理**へ改善する必要があります。

改善対象の例：

* tenant policy
* channel policy
* sender profile
* thread summary
* current message
* open actions

要するに、**主Agentには高信号の要約だけを渡し、重い検索や全文処理は子Agentや外部処理へ分離する**方向が必要です 

---

### 3. 確率的判断の外側に、決定的な制御面を追加

分類・要約・検索のような読み取り系処理と、送信・更新・登録のような変更系処理を同列に扱わないことが重要です。
特に改善が必要なのは以下です。

* 権限分離
* Policy Gate
* Human Gate
* approval 前 hook
* prompt injection 対策
* sandbox / allowlist
* 高リスク操作の明示的承認

つまり、**「AIに慎重に判断させる」ではなく、「危険操作を構造的に止める」設計に改善すること**が必要です 

---

### 4. connector / tool を標準化

Slack、WhatsApp、Email、CRM、KB などの接続を if/else 的に埋め込むのではなく、**契約ベースの標準化された connector / tool 設計**に改善する必要があります。

改善観点：

* canonical schema
* auth scope
* idempotency contract
* error surface
* 明確な tool 定義
* 狭く意味の明確な API

たとえば万能の `http_request` や曖昧な `memory_write` ではなく、
`lookup_sender_profile(...)` や `create_reply_draft(...)` のような、**狭く・強い意味を持つツール**へ改善するのが望ましいです 

---

### 5. 単一 orchestrator から focused subagents 構成へ改善

1つの orchestrator が分類、返信作成、メモリ更新、要約まで全部担う構成は、デモとしては成立しても、責務分離が弱くなります。
そのため、以下のような **focused subagents** への分割が改善要求になります。

* Triage Subagent
* Reply Draft Subagent
* Summary Subagent
* Action Router
* Policy / Hook chain

ポイントは、**いきなり複雑な agent teams にするのではなく、まずは責務の明確な subagent 分割に留めること**です 

---

### 6. replay / eval / monitoring を本体機能として組み込む

「とりあえず動く」では不十分であり、以下を持つ必要があります。

* replay 可能な履歴
* pre-send checkpoint
* failure replay
* CI 回帰評価
* drift 監視
* unsafe action rate の測定
* reply acceptance / escalation latency などの指標管理

つまり、**改善要求は機能追加だけではなく、変更の影響を検証できる評価基盤の整備まで含みます**

## 段階分解

### Stage 1. 実行基盤の一等化

- execution session / decision / checkpoint / feedback / action log を task 単位で保存する
- task detail, execution, replay の read-only API を整備する
- orchestration の開始、補足入力、route、verification、完了/失敗を substrate に落とす

### Stage 2. 階層化コンテキスト

- global policy / session summary / request context / open actions / retrieval evidence refs を分離する
- main agent には高信号 summary のみ渡す
- specialist / verifier には必要最小限の context を分けて注入する

### Stage 3. Focused Subagents 化

- coordinator の責務を triage / action-router / summary に分離する
- task template ごとに handler を直接分岐するのではなく、focused task plan を介して route する
- route metadata を結果と replay で追えるようにする

### Stage 4. Connector / Tool 契約標準化

- template ごとに semantic tool contract を定義する
- contract には action name, auth scope, idempotency key, error surface, backing skill を持たせる
- generic tool 名ではなく、意味の明確な action 名で inspection できるようにする

### Stage 5. Control Plane 強化

- semantic contract を基準に read / analyze / draft / write を分離する
- read_only / approval_required / autonomous を deterministic に評価する
- mutation 系は pre-write checkpoint / human gate / structural stop を適用する

### Stage 6. Replay / Eval / Monitoring 常設化

- replay に substrate records だけでなく execution events も束ねる
- verifier / scorer / approval を横断した monitoring summary を提供する
- unsafe action rate, verification failure rate, quality score, approval latency を継続観測可能にする

### Stage 7. 回帰テスト固定化

- stage ごとに unit / API テストを追加する
- orchestration, coordinator, replay, monitoring, control plane の回帰点を固定する
- `check.sh` と pytest の対象を段階的に広げる
