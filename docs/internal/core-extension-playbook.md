# Core 拡張プレイブック

## 1. Core 変更の前提

Core を変更するのは以下を満たす場合のみ:

1. 複数製品線で共通に必要
2. Plugin では責務が過剰分散する
3. 安定契約として長期維持できる

## 2. 実施フロー

1. 目的と境界を RFC 化
2. 既存 Plugin で代替不能か検証
3. API 契約を先に固定
4. 実装
5. 回帰テスト
6. ドキュメント更新

## 3. 禁止事項

- 便宜上の理由で機能を Core に取り込む
- 製品固有ロジックを Core へ追加する
- 監査経路を迂回する

## 4. 推奨チェックリスト

1. 型契約に破壊変更がないか
2. 監査情報が欠落していないか
3. Studio/Framework で判定差分が仕様どおりか
4. 旧経路が残っていないか
